Yes. Snapshots solve the provisioning speed problem.

A fully provisioned inference VM — OS booted, drivers loaded, model server running, model weights loaded into VRAM — takes minutes to create from scratch. But a snapshot of that state takes seconds to restore.

Snapshot-based model provisioning:

First deployment — full provision: LP creates VM, NixOS boots, CUDA loads, model server starts, weights pull from source (slow — minutes for large models). This happens once.

Snapshot — after health check passes (model loaded, serving, latency acceptable), Weaver snapshots the entire VM state: disk + memory. Model weights are already in VRAM. The snapshot captures a running, warm inference VM.

Subsequent deployments — restore from snapshot. VM resumes with model already loaded in VRAM. Seconds, not minutes. This is what makes auto-scaling (Decision #95) viable for inference: preProvisionAt triggers → restore snapshot → VM is serving in seconds → enters bridge rotation.

Blue/green — snapshot the new model version after health check. If rollback needed later, restore from the previous version's snapshot. Both versions exist as snapshots. Rollback is instant.

What gets snapshotted:

Disk snapshot — OS + model weights on disk + model server config. This is the reproducibility layer — NixOS flake lock guarantees the environment is identical.
Memory snapshot (optional, QEMU only) — RAM + VRAM state. Model is already loaded. Resume = warm start. Firecracker doesn't support memory snapshots. QEMU does. This is another reason GPU inference VMs use QEMU, not Firecracker.
Snapshot library in Shed:

This merges with the model library concept. A model in the library can have associated snapshots:

Model reference (pointer to weights + config) — what to deploy
Template (NixOS flake + model server + GPU config) — how to deploy it
Snapshot (disk + optional memory state) — pre-built, ready to restore
The Shed creation flow becomes:

No snapshot exists → full provision (minutes) → auto-snapshot on successful health check
Snapshot exists → restore (seconds) → health check → serve
Snapshot lifecycle:

Snapshots are invalidated when the template changes (new CUDA version, new model server version, new NixOS flake lock)
Snapshots are version-tagged alongside model version
Old snapshots pruned by policy (keep last N versions, or keep all production snapshots)
Snapshot storage is local per-host. At Fabrick scale, the hub tracks which hosts have which snapshots — feeds into fleet scheduling (Decision #116): prefer host with cached snapshot over host that needs full provision
Tier split:

Capability	Tier	Version
Auto-snapshot on successful inference health check	Weaver	v2.0 (ships with template catalog + snapshots)
Restore from snapshot in Shed	Weaver	v2.0
Snapshot-based auto-scaling (restore on set point trigger)	Fabrick	v2.2 (Decision #95 + snapshots)
Fleet snapshot awareness (which hosts have which snapshots)	Fabrick	v3.0
Fleet scheduling prefers snapshot-cached hosts	Fabrick	v3.0
The speed comparison:

Method	Cold start time	Warm (model in VRAM)
Full provision (pull weights, load model)	3–10 min (70B model)	No
Disk snapshot restore	10–30 sec	No (model reloads from disk)
Memory snapshot restore (QEMU)	2–5 sec	Yes — model already in VRAM
Memory snapshot restore is the killer feature. The inference VM resumes exactly where it was — model loaded, VRAM warm, ready to serve. This is what makes Firecracker-speed auto-scaling possible for GPU inference on QEMU.
