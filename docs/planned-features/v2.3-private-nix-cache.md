## Feature summary

**Private Nix Cache via Attic (Decision #147):**
- Self-hosted Nix binary cache (`zhaofengli/attic`) as managed Weaver workload
- Multi-tenant caches (one per Team peer group at v2.3; fleet-scope expansion at v2.4)
- Approved Packages allowlist with manual admin approval at Solo/Team entry
- Add-only signing-key rotation (rotate / retire / compromise operations)
- Peer-federated allowlist replication across Team peers
- Runtime substituter writer locks `/etc/nix/nix.conf` to the private cache
- Tombstone revocation preserves audit trail

**Shed Builder custom software ingestion (Decision #149):**
- Product-internal Hydra-pattern build service (not an import of Hydra itself, not a reuse of Forge)
- Four ingestion lanes: Lane 1 binary drop (Solo+, v2.3), Lane 2 OCI/Dockerfile import (Solo+, v2.3), Lane 3 git-repo build (Team+, v2.4), Lane 4 native Nix derivation (Solo+, v2.3)
- Compliance attestation schema (`attestation` JSONB + `ingestion_source` enum) on `approval_records` table
- Two-person rule at Team+ for Lane 1 (configurable default-on with audit-logged disablement)

## Target control mappings

The v2.3 feature maps to these compliance controls. Each has been verified against the canonical control text.

### NIST 800-171 Rev 2

- **3.4.8 — Apply deny-by-exception (blacklist) policy to prevent use of unauthorized software or deny-all, permit-by-exception (whitelist) policy to allow the execution of authorized software.** Exact match: Approved Packages IS a whitelist-based software execution policy.

### HIPAA §164.312 Technical Safeguards

- **§164.312(c)(1) Integrity — Implement policies and procedures to protect electronic protected health information from improper alteration or destruction.** Good fit: signed approved-package cache prevents unauthorized software tampering with ePHI-processing systems.

### PCI DSS v4.0

- **Req 6.4.5 — Changes to critical software are documented and approved.** Multi-faceted fit: approval records provide the documentation + approval chain.
- **Req 11.5.1 — Deploy change-detection mechanism to alert personnel to unauthorized modification of critical files.** Preventive FIM: signed package cache rejects unapproved derivation hashes at install time.

### SOC 2 (AICPA 2017 TSC)

- **CC6.8 — The entity implements controls to prevent or detect and act upon the introduction of unauthorized or malicious software to meet the entity's objectives.** Exact match: CC6.8 is directly about preventing unauthorized/malicious software introduction.

### CIS Controls v8.1

- **Control 2 — Inventory and Control of Software Assets.** Exact match: approved-package allowlist with audit log IS the software inventory + control mechanism.
  - **Safeguard 2.1** — Establish and maintain a software inventory
  - **Safeguard 2.3** — Address unauthorized software (remove or document as exception)
  - **Safeguard 2.5** — Allowlist authorized software
  - **Safeguard 2.6** — Allowlist authorized libraries (v3.1 SBOM extraction)
  - **Safeguard 2.7** — Allowlist authorized scripts

### 21 CFR Part 11 (pharma, no current mapping doc)

- **§11.10(d) — Limiting system access to authorized individuals + §11.10(e) Secure computer-generated audit trails.** Good fit for Approved Software feature when a Part 11 mapping doc is eventually created.
