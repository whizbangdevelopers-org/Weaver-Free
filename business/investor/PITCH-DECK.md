## Slide 2: The Problem

### The entire IT infrastructure industry is flying blind.

Every management tool today — Proxmox, VMware, Nutanix, Kubernetes — is built on mutable infrastructure. You deploy. Then the system changes underneath you.

| What we think we have | What we actually have |
|---|---|
| ✓ All healthy | ⚠ Config drifted |
| ✓ Configs applied | ⚠ Manual SSH fixes |
| ✓ Compliant | ✗ State mutated |
| ✓ Up to date | ✗ Undocumented pkgs |

- **Configs drift.** Packages mutate. Manual fixes accumulate.
- **Every tool shows you what you deployed** — not what's actually running.
- **Audit evidence?** Hope someone wrote it down.
- **Every compliance scan** is a point-in-time snapshot of a moving target.

The infrastructure you see is not the infrastructure you have.

The fix isn't better monitoring on a drifting foundation. It's a foundation where drift is architecturally impossible. That foundation exists. What it doesn't have is a management layer.

> **Speaker note:** This isn't a NixOS problem. This is every server, every VM, every container in every data center built on mutable infrastructure. Pause here — let them feel it. The closer teases the solution without naming NixOS yet.

## Slide 3: The Solution

### Weaver is the foundation the management layer never had.

That foundation is NixOS — the only OS where configuration drift is mathematically impossible. Declarative, reproducible, immutable. Roots back to 2003, shipping stable releases for 12 years, 100K+ packages in production at ~466 companies. This isn't experimental — it's mature infrastructure that never had a management layer.

No NixOS expertise required. But until now, no one built the operational layer on top.

Weaver is a triple-interface management platform — web, terminal, and mobile. Same backend, same API, same tier gating. Manage your fleet from your desk, your terminal, or your pocket.

**The pitch:** Deploy Weaver alongside your existing Docker, Proxmox, or VMware stack. No migration, no cutover, no risk. MicroVMs run in hardware isolation alongside your containers — Weaver manages both from one pane. NixOS is the implementation detail. The value is control.

**Core capabilities:**
- **Web interface** — Quasar PWA with real-time WebSocket updates, VM lifecycle, Strands (local topology), AI diagnostics
- **Terminal UI** — React/Ink TUI with 97% web feature parity, for CLI-native admins who live in the terminal

- **Live Provisioning** — create and manage VMs via API/UI without ever running `nixos-rebuild switch` on the host. One-time NixOS setup, then zero host rebuilds forever.
- **AI operations** — Free: BYOK (bring your own API key); Weaver Solo+: admin-managed AI credential vault; natural language diagnostics, streaming responses

**The key insight:** Parallel migration — Weaver runs *alongside* existing infrastructure. No rip-and-replace. Admins evaluate it on their own hardware, prove value, then champion it internally. NixOS's declarative model means the running state of every VM is always exactly what's declared. Audit trails, compliance evidence, and reproducible recovery come free.

## Slide 10: The Microservices Opportunity

### The industry adopted microservices for the architecture and got Kubernetes as the tax.

**The trend:** Microservices architecture dominates enterprise software — independent deployment, isolated failure domains, per-service scaling. But the operational reality is Kubernetes, service mesh, platform teams, and $500K+/yr in infrastructure overhead.

**The backlash is real:**
- Platform engineering exists as a discipline because K8s is too complex for app teams to operate
- CNCF surveys: "complexity" is the #1 K8s adoption barrier, every year
- Median enterprise K8s deployment: 3–5 full-time platform engineers ($450K–$1.25M/yr)
- CNI + ingress + Argo Rollouts + service mesh = 4 components with 4 failure surfaces

**Weaver's answer: microservice benefits, zero orchestration tax.**

| Microservice Benefit | Kubernetes Delivers Via | Weaver Delivers Via |
|---|---|---|
| Independent deployment | Helm charts + pod rolling updates | LP API call — Firecracker boots in seconds |
| Isolated failure domains | Namespace isolation (shared kernel) | MicroVM hardware isolation (separate kernel) |
| Per-service scaling | HPA + pod autoscaler | Bridge weight shifting + snapshot restore (2–5 sec) |
| Blue/green deployments | Argo Rollouts + service mesh | Smart Bridges — AI-managed, one component |
| Zero-drift config | GitOps (ArgoCD/Flux) | NixOS — drift is impossible by design |

A competitor would need four prerequisites to match this: (1) a multi-hypervisor abstraction layer, (2) a benchmarking framework with network-isolated measurements, (3) a declarative system for reproducible VM definitions, (4) per-workload metrics infrastructure for overhead comparison. Weaver has all four. A competitor starting from scratch needs years.

*"Pick the right isolation backend for each workload — Weaver proves which one is best on your hardware."*

**The zero-drift moat:** NixOS makes configuration drift impossible by construction. No scanning tools, no remediation workflows, no "who changed this?" fire drills. The running state is always exactly what's declared. This is a structural advantage no imperative platform can replicate.

**The post-Glasswing moat:** Every competitor in this table runs workloads on shared kernels. Project Glasswing demonstrated that AI finds kernel-level vulnerabilities faster than humans can patch them. VMware, Proxmox, Docker, and Kubernetes all share the host kernel across workloads — a single AI-discovered zero-day compromises every tenant. Weaver's MicroVMs run separate kernels with hardware isolation. Container escapes are not a risk to mitigate — they are structurally impossible. This is the only architecture in the comparison built for a world where zero-days are discovered at machine speed.

**The Observer bridge:** Competitors require full commitment before any visibility. FabricK lets enterprises observe their entire existing fleet (VMware, Proxmox, bare Linux) from day one via `weaver-observer` — a Rust binary that runs on any Linux. Convert hosts incrementally. No rip-and-replace. The "Convert to Managed" button on every Observed host is the upgrade trigger.

**"Does this need more hardware?"** No. MicroVMs add 50–128MB of RAM overhead per guest for the isolated kernel — on a 256GB host, that's less than 5% of capacity. What you get back is hardware-grade isolation that makes compliance automatic and eliminates container-escape vulnerabilities entirely. And Weaver manages Docker and Podman containers on the same host, same interface, same Smart Bridges — if a workload doesn't need hardware isolation, run it as a container. The admin chooses the isolation level per workload, not per platform. The TCO math already accounts for this: $12K/yr replaces $175K/yr regardless of the RAM overhead.

**Observer distribution:** `weaver-observer` ships as a statically-linked Rust binary (zero runtime dependencies) in 4 channels: signed binary + install script, `.deb` package, `.rpm` package, and Ansible Galaxy role. Two architectures: x86_64 (servers, cloud) and aarch64 (edge, ARM, Graviton). Install on any Linux host in under 60 seconds.

**Technology alliances:** Confirmed integrations with 1Password (SSO/credentials), Yubico (FIDO2/WebAuthn), Kanidm (identity). Planned: Tailscale (peer discovery), NVIDIA (GPU scheduling), Hetzner + DigitalOcean (cloud workloads), Anthropic (AI operations BAA for HIPAA).
