Weaver / FabricK Score Justification

Security & Isolation

1
3Default security posture. NixOS is immutable by default — no mutable system state to compromise. Declarative firewall means nothing is open unless explicitly declared. Every MicroVM gets its own kernel (hardware isolation), not a shared-kernel container with namespace boundaries. No default credentials — first-run admin setup required. The Insurance Principle ensures Free tier ships zero security features and zero attack surface from security tooling. Paid tiers add hardening extensions (AppArmor, Seccomp, lanzaboote Secure Boot). OpenShift scores 3 for SCCs and SELinux — but those are namespace-level controls on a shared kernel. W/F’s default is hardware isolation per workload on an immutable OS.
2
3RBAC granularity. Per-VM ACL (FabricK) provides per-resource access control on actual workloads — more granular than K8s namespace-level boundaries. Workload Groups (v3.3) create compliance-aligned scope boundaries that map 1:1 to network boundaries. IdP group sync auto-provisions access from AD/LDAP. Scoped Weaver views ensure users only see what they’re authorized to see. Per-workload AI vendor enforcement gives admins control over which AI touches which workload. K8s RBAC is verb-level granularity on API calls (get/list/watch/delete) — plumbing granularity. W/F’s RBAC is access control on the things operators actually care about: workloads, compliance boundaries, and identity.
3
3Network policy enforcement. Fleet virtual bridges are overlay-backed (VXLAN/WireGuard) and workload-group-aligned — network boundary = compliance boundary, 1:1. Smart Bridges provides AI-operated routing: health-based, blue/green, weight shifting, inference routing. Managed bridges with IP pools at Weaver Solo+. Firewall extensions (v1.2) deliver presets, custom rules, and egress control at Weaver; zones, drift detection, and audit at FabricK. Network Isolation Mode (v1.3) for explicit isolation. Each MicroVM has its own kernel and network stack — hardware-isolated by default. K8s scores 2 because it needs a third-party CNI (Calico, Cilium) bolted on just to get network policies — it ships with none enforced by default.
4
3Secrets management. sops-nix provides encrypted-at-rest secrets with KMS support (FabricK). SQLCipher encrypted credential vault (v1.5) with opaque encrypted blob storage — vault doesn’t assume API key shape. AI credential vault (v1.4) for admin-managed frontier model keys and application-specific AI credentials. Per-workload credential assignment with full credential audit trail. Fleet vault federation (v3.0) — credential added to one Weaver node becomes optionally available fleet-wide via FabricK control plane. K8s secrets are base64-encoded by default — not even encrypted. You need to bolt on external Vault or enable etcd encryption separately.
5
3Runtime isolation (per task). Every MicroVM gets its own kernel — hardware isolation, not namespace isolation on a shared kernel. This is W/F’s single biggest structural advantage over every container orchestrator. A compromised workload cannot affect other workloads because there is a hardware boundary, not a software boundary, between them. Containers (Docker/Podman/Apptainer) are also managed for workloads that don’t need hardware isolation. K8s scores 1 — all pods share a kernel. ECS Fargate scores 3 via Firecracker microVMs. No other orchestrator offers hardware isolation as the default.
6
3Supply chain controls. Nix’s content-addressed store provides bit-for-bit reproducible builds — every dependency is pinned by hash, not by mutable tag. No “latest” tag, no mutable layers, no registry that can be poisoned after the fact. Nix flake lock files pin every input. Attic binary cache (FabricK v2.3) provides fleet-wide Nix store deduplication. License audit CI gate blocks non-compliant licenses (e.g., Redis SSPL → Valkey BSD-3). Container registries are fundamentally mutable — “ubuntu:22.04” can change between pulls. Nix hashes cannot.
7
3Attack surface. NixOS minimal host — only declared packages are present, no leftover binaries from previous installs. MicroVM guests have minimal attack surface by construction. Hardening extensions (AppArmor, Seccomp profiles, kernel hardening) at Weaver tier. Lanzaboote Secure Boot at FabricK. No shell in production by default — NixOS doesn’t need a package manager at runtime. K8s scores 1 — the API server, etcd, kubelet, kube-proxy, and CNI plugin all present attack surface. Nomad scores 3 for its single-binary simplicity.

Compliance & Auditability

8
3Audit logging. All auth events logged (login, logout, register, password change, role change, user delete). Credential audit trail on every vault operation. Compliance Export Extension covers 8 regulatory frameworks (HIPAA, SOC 2, PCI-DSS, NIST 800-171, CIS Benchmarks, etc.) with per-control tier qualification. Group change audit trail (Compliance Pack, v3.3). Purpose-built compliance audit with exportable reports — not “install a sidecar and parse JSON logs.”
9
3Declarative config. Everything is Nix — host configuration, workload definitions, network topology, firewall rules, secrets, and service composition are all declarative, all in the same language, all version-controlled. Live Provisioning creates VMs via API without nixos-rebuild switch — runtime dynamism on a declarative foundation. Host Config viewer shows the full declarative state in the UI. K8s YAML is declarative but spread across dozens of resource types, tools, and CRDs. NixOS is one language, one evaluation, one system.
10
3Config drift detection. NixOS makes drift structurally impossible — the running system is a function of the declared configuration. There is no mutable state that can drift. FabricK adds config sync across the fleet (Colmena fleet deploy), ensuring all nodes converge to the declared state. Firewall drift detection and audit at FabricK tier. K8s drift detection requires GitOps tooling (ArgoCD, Flux) bolted on after the fact. NixOS prevents drift by construction.
11
3Reproducible deployments. Nix flake distribution provides bit-for-bit reproducible builds. Every dependency is content-addressed and pinned. nix build on any machine with the same flake.lock produces the same output. NixOS generations provide instant rollback to any previous system state. Snapshot provisioning enables “build once, run many” with 2–5 sec VRAM-warm restore. Container images are “reproducible-ish” — same Dockerfile can produce different images depending on when you build. Nix is deterministic by construction.
12
2Compliance certifications. No formal FedRAMP or SOC 2 certification yet. However: Compliance Export Extension maps 8 regulatory frameworks at the section level with per-control tier qualification. Defense contractor, government, healthcare, financial services, pharma, and education vertical sales docs provide regulation-specific mapping. Full audit trail + encrypted secrets + RBAC + reproducible builds provide the posture. K8s at 2 also lacks certifications — managed services (EKS, GKE) inherit cloud provider certifications. W/F’s readiness posture matches K8s without the cloud dependency.
13
2FedRAMP / FISMA readiness. Strong readiness posture without a formal Authority to Operate. NixOS reproducibility, full audit logging, encrypted secrets (sops-nix/KMS), RBAC, and compliance export provide the technical controls. Defense contractor and government vertical docs map specific regulations. K8s at 1 has no inherent FedRAMP readiness. ECS at 2 inherits AWS’s ATO. OpenShift at 3 has Red Hat’s certifications. W/F’s controls exceed bare K8s; the gap is the formal stamp.
14
3GitOps native support. NixOS configuration IS the git repo. Every change to the system is a commit. Nix flake inputs are pinned references to git repositories. Colmena fleet deploy pushes git-committed configurations to the entire fleet. No ArgoCD, no Flux, no reconciliation loop needed — the system is the repo. K8s GitOps requires dedicated tooling (ArgoCD/Flux) to reconcile declared state with cluster state. NixOS has no gap to reconcile.

AI-Augmented Threat Response

Context: Anthropic’s Project Glasswing (April 2026) — a coalition with AWS, Google, Microsoft, NVIDIA, CrowdStrike, and others — uses frontier AI (Claude Mythos Preview) to find and fix vulnerabilities at scale. The project has already discovered thousands of zero-day vulnerabilities, including some that survived decades of human review, across every major operating system and browser. Anthropic committed $100M in usage credits and stated: “AI cyber capabilities at this level will proliferate over the coming months, and not every actor who gets access to them will be focused on defense.” This category evaluates how each orchestration platform responds to an era where zero-days are discovered faster than humans can patch them.

30
3Zero-day blast radius. When AI discovers a kernel zero-day, every workload sharing that kernel is compromised simultaneously. Weaver’s hardware boundary per MicroVM means a kernel zero-day compromises one workload, not your entire fleet. Five hypervisor options (QEMU, Cloud Hypervisor, crosvm, kvmtool, Firecracker) mean a hypervisor-specific vulnerability doesn’t affect workloads running on a different hypervisor — defense through diversity. NixOS immutability means even if the host is reached, there’s no mutable state to persist a rootkit. K8s scores 0 — all pods share the host kernel. A single kernel zero-day, exactly the kind AI is now finding by the thousands, compromises every pod on every node simultaneously. Docker Swarm scores 0 for the same reason. ECS Fargate scores 2 via Firecracker microVMs but is locked to a single hypervisor. Nomad scores 1 — it can schedule VMs via QEMU driver but defaults to containers on a shared kernel.
31
3Patch propagation speed. When Glasswing-class tools discover vulnerabilities, the fix must propagate everywhere — fast, verifiable, auditable. NixOS’s flake.lock pins every dependency by hash. Pin the patched version, rebuild, deploy via Colmena (FabricK) — every node converges to the fixed state deterministically. No “did we patch that server?” No config drift. The entire dependency closure is content-addressed, so you can prove the fleet is patched. K8s scores 1 — Helm/Kustomize can roll out updated images, but mutable registries and layer caching mean there’s no deterministic guarantee the same image runs everywhere. ECS scores 2 for task definition versioning with ECR immutable tags. Nomad scores 1 for job versioning without deterministic builds. Docker Swarm scores 0 — no fleet-wide deployment guarantee and no config management story.
32
3Supply-chain verifiability. Glasswing’s scope explicitly includes “open-source and supply-chain security” and “software development lifecycle and secure-by-design practices.” NixOS’s content-addressed store makes the entire supply chain formally verifiable — every package is identified by its complete dependency tree hash, not a mutable tag. nixpkgs (100,000+ packages) is the largest curated package repository in the world. License audit CI gate formally verifies every dependency license. K8s scores 1 — container scanning tools exist (Trivy, Snyk) but the underlying model is mutable registries and “latest” tags. ECS scores 1 for ECR scanning but same mutable registry model. OpenShift scores 1 for Red Hat’s certified catalog but still container-based. Docker Swarm and Nomad score 0 — no built-in supply chain verification.
33
3Hypervisor diversity. Monoculture is the enemy of security — a single vulnerability in a single runtime shouldn’t compromise an entire fleet. Weaver supports 5 hypervisors (QEMU, Cloud Hypervisor, crosvm, kvmtool, Firecracker), each with a different codebase and attack surface. Operators can run different hypervisors for different workload classes, ensuring that an exploit discovered in one (even by AI-augmented attackers) doesn’t cascade. Every orchestrator in this comparison is locked to a single container runtime (containerd, Docker Engine, or rkt). K8s scores 0 — containerd/CRI-O, both shared-kernel runtimes. ECS scores 0 — locked to Fargate’s Firecracker or EC2’s Docker. Nomad scores 0 — Docker driver only for containers. A discovered vulnerability in that single runtime compromises every workload on the platform.
