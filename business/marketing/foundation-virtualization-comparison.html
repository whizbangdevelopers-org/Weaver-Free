Weaver / FabricK Score Justification

Security & Isolation

1
3Kernel isolation. Inherits NixOS MicroVM’s hardware-isolated kernel per workload and adds managed security layers on top: 3-role RBAC (admin/operator/viewer), per-VM ACL at FabricK tier, tier gating via requireTier() middleware, and rate limiting gradient (5/10/30 by tier). Every workload runs in its own kernel — the same hardware boundary as a full VM but with MicroVM efficiency. Docker and nspawn score 0 — all containers share the host kernel. Weaver manages both MicroVMs (hardware-isolated) and containers (namespace-isolated) uniformly, giving operators the choice per workload.
2
3Attack surface. NixOS minimal host — only declared packages exist, no leftover binaries from previous installs, no package manager at runtime. MicroVM guests are minimal by construction. Weaver adds hardening extensions at paid tiers: AppArmor profiles, Seccomp filtering, kernel hardening at Weaver; impermanence (ephemeral root) and lanzaboote Secure Boot at FabricK. The Insurance Principle ensures Free tier exposes zero security surface from tooling. Full VMs (QEMU/Proxmox) score 1 — full guest OS with all the attack surface that implies. Firecracker scores 3 for its minimal VMM design. W/F combines minimal VMM with minimal host OS.
3
3VM/container escape risk. Hardware boundary between every MicroVM workload — escaping requires a hypervisor exploit, not a kernel exploit. Weaver supports 5 hypervisors (QEMU, Cloud Hypervisor, crosvm, kvmtool, Firecracker), each with different attack surfaces — a vulnerability in one doesn’t compromise workloads on another. NixOS immutability means even if the host is reached, there’s no mutable state to persist a rootkit. Docker scores 1 — container escapes are a well-documented, regularly exploited vulnerability class because all containers share the host kernel.
4
3Covert channel resistance. Six covert channel mitigations exposed as toggleable hardening profiles per MicroVM or per Workload Group (v1.2 hardening extensions): (1) CPU core pinning — dedicate physical cores per MicroVM, eliminating shared cache side channels. (2) Intel CAT / AMD cache partitioning — hardware cache isolation between VMs. (3) Speculative execution hardening — enforce all mitigations (retpolines, IBRS, STIBP, SSBD) as a toggleable profile. (4) Intel MBA — isolate memory bus bandwidth per VM. (5) Network traffic padding via Smart Bridges (FabricK) — constant-rate shaping on fleet virtual bridges to defeat traffic analysis. (6) Timer resolution reduction — reduce guest timer precision via hypervisor config. Items 1–4 are hardware-level isolation that no shared-kernel container solution can match. Required for defense/HIPAA/financial compliance (NIST 800-53 SC-4, DISA STIG). Unikernels score 3 for minimal guest surface. W/F matches with hardware-level partitioning (CPU, cache, memory bus, network, timers) on top of hardware-isolated VMs — a deeper mitigation stack than any other virtualization platform in this comparison.
5
3Syscall surface control. NixOS MicroVM guests expose only the syscalls the guest kernel supports — no host kernel syscall surface shared with workloads. Weaver adds Seccomp profile extensions and AppArmor mandatory access control at paid tiers, providing declarative syscall filtering on top of the hardware boundary. FabricK adds kernel hardening and Secure Boot. gVisor also scores 3 for its user-space kernel that intercepts and filters every syscall — a different approach (software interposition vs. hardware boundary) with similar effectiveness. Docker scores 1 — default Seccomp profile is permissive; most deployments don’t customize it.

Reproducibility & Compliance

6
3Declarative config. Everything is Nix — host configuration, MicroVM definitions, container definitions, network topology, firewall rules, secrets, and service composition. All in one language, all version-controlled, all evaluatable offline. Live Provisioning adds runtime dynamism (create VMs via API) on the declarative foundation — no nixos-rebuild switch required. Host Config viewer shows the full declarative state categorized by workload type in the UI. Docker scores 1 — Dockerfiles are imperative build scripts, not declarative system descriptions. Firecracker and gVisor score 0 — they are runtimes with no configuration management story.
7
3Reproducible builds. Nix’s content-addressed store pins every dependency by hash. nix build on any machine with the same flake.lock produces the same output, bit-for-bit. No “latest” tags, no mutable layers, no registry that can change between pulls. NixOS generations provide instant rollback to any previous system state. Attic binary cache (FabricK v2.3) provides fleet-wide Nix store deduplication. Docker scores 1 — same Dockerfile can produce different images depending on build time, mirror state, and layer cache. Unikernels score 2 for their build-system reproducibility but lack the ecosystem-wide pinning Nix provides.
8
3Git-auditable state. NixOS configuration IS the git repo — every change to the system is a commit with full diff history. Nix flake inputs are pinned git references. FabricK adds audit logging on all auth events, credential operations, and workload changes. Config export (v1.6) produces machine-readable system snapshots. Colmena fleet deploy pushes git-committed configurations to the entire fleet. Full VMs score 1 — VM state is opaque disk images with no inherent version control. Docker scores 1 — Dockerfiles are auditable but running container state is not.
9
3Config drift protection. NixOS makes drift structurally impossible — the running system is a pure function of the declared configuration. There is no mutable state that can accumulate. FabricK adds fleet-wide config sync via Colmena, ensuring all nodes converge to the declared state. Firewall drift detection and audit at FabricK tier catches any runtime rule modifications. Docker scores 1 — running containers can be exec’d into and mutated; no mechanism prevents drift from the image. Full VMs score 0 — VMs are inherently mutable.

Operations

12
3Boot time. Live Provisioning creates and starts VMs via API without nixos-rebuild switch — instant workload creation, not a rebuild cycle. MicroVMs boot in sub-second to low-single-second range depending on hypervisor. Snapshot provisioning delivers 2–5 sec restore from VRAM-warm memory snapshots — “build once, run many.” Bare NixOS MicroVM scores 2 because without Live Provisioning, creating a new VM requires editing configuration.nix and running nixos-rebuild. Weaver eliminates that bottleneck. Docker and Firecracker score 3 for sub-second cold start.
13
2Memory efficiency. Same as the underlying NixOS MicroVM runtime — Weaver is a management layer, not a runtime modification. MicroVMs allocate dedicated RAM per guest (no memory ballooning by default in most hypervisors). More efficient than full VMs (no full OS overhead per guest) but less efficient than containers (which share the host kernel’s memory). Docker and nspawn score 3 — shared kernel means near-zero per-container memory overhead. Firecracker scores 3 for its minimal VMM footprint. This is a fundamental tradeoff: hardware isolation costs memory. W/F doesn’t hide that.

15
3Live migration. Three complementary capabilities for zero-downtime workload management. Continuous fleet load balancing (Smart Bridges, v2.2+): weight shifting distributes traffic across instances; health-based routing avoids overloaded or unhealthy nodes before they degrade; inference routing directs requests to instances with the right model loaded; auto-scale adds capacity on demand. Prevents overload rather than reacting to it. Workload Group-level migration (Smart Bridges): blue/green and cordon/drain operate at the bridge level — bridges are 1:1 with Workload Groups, so entire groups of workloads migrate simultaneously as a unit, not one VM at a time. Snapshot provisioning delivers 2–5 sec VRAM-warm restore on any fleet node — no shared storage, no memory copy overhead, works across cloud providers. Traditional VM live migration (FabricK v3.0): memory-state migration between fleet nodes with HA for workloads requiring stateful continuity. Full VMs (QEMU/Proxmox) score 3 for mature single-VM memory-state migration — one mechanism that moves one VM at a time after it’s already overloaded. AI-Augmented Threat Response

Context: Anthropic’s Project Glasswing (April 2026) — a coalition with AWS, Google, Microsoft, NVIDIA, CrowdStrike, and others — uses frontier AI (Claude Mythos Preview) to find and fix vulnerabilities at scale. The project has already discovered thousands of zero-day vulnerabilities, including some that survived decades of human review, across every major operating system and browser. Anthropic committed $100M in usage credits and stated: “AI cyber capabilities at this level will proliferate over the coming months, and not every actor who gets access to them will be focused on defense.” This category evaluates how each virtualization technology responds to an era where zero-days are discovered faster than humans can patch them.

20
3Zero-day blast radius. When AI discovers a kernel zero-day, every workload sharing that kernel is compromised simultaneously. Weaver’s hardware boundary per MicroVM means a kernel zero-day compromises one workload, not your entire fleet. Five hypervisor options (QEMU, Cloud Hypervisor, crosvm, kvmtool, Firecracker) mean a hypervisor-specific vulnerability doesn’t affect workloads running on a different hypervisor — defense through diversity. NixOS immutability means even if the host is reached, there’s no mutable state to persist a rootkit. Docker scores 0 — all containers share the host kernel. A single kernel zero-day, exactly the kind AI is now finding by the thousands, compromises every container on the host simultaneously. This is no longer a theoretical risk; it’s the documented output of Project Glasswing.
21
3Patch propagation speed. When Glasswing-class tools discover vulnerabilities, the fix must propagate everywhere — fast, verifiable, auditable. NixOS’s flake.lock pins every dependency by hash. Pin the patched version, rebuild, deploy via Colmena (FabricK) — every node converges to the fixed state deterministically. No “did we patch that server?” No config drift. The entire dependency closure is content-addressed, so you can prove the fleet is patched. Glasswing’s 90-day public disclosure cycle means organizations must demonstrate they can patch at the speed of AI-discovered vulnerabilities. Docker scores 1 — rebuild images, push to registry, restart containers, hope nothing mutated at runtime. Full VMs score 0 — each VM is an opaque disk image requiring individual patching with no deterministic guarantee.
22
3Supply-chain verifiability. Glasswing’s scope explicitly includes “open-source and supply-chain security” and “software development lifecycle and secure-by-design practices.” NixOS’s content-addressed store makes the entire supply chain formally verifiable — every package is identified by its complete dependency tree hash, not a mutable tag. nixpkgs (100,000+ packages) is the largest curated package repository in the world, with maintainers already participating in the Glasswing-funded OpenSSF and Alpha-Omega programs. License audit CI gate formally verifies every dependency license. Docker scores 1 — “latest” tags, mutable layers, and registry-side changes mean the same Dockerfile can produce different images depending on when and where you build. Supply-chain attacks targeting container registries are a well-documented vulnerability class.
23
3Hypervisor diversity. Monoculture is the enemy of security — a single vulnerability in a single hypervisor shouldn’t compromise an entire fleet. Weaver supports 5 hypervisors (QEMU, Cloud Hypervisor, crosvm, kvmtool, Firecracker), each with a different codebase and attack surface. Operators can run different hypervisors for different workload classes, ensuring that an exploit discovered in one (even by AI-augmented attackers) doesn’t cascade. This is the same principle behind genetic diversity in biological systems: monocultures are fragile, diversity is resilient. No other technology in this comparison offers hypervisor choice. Full VMs score 1 for QEMU/KVM only. Firecracker, gVisor, nspawn, and Qubes are each locked to a single runtime — a discovered vulnerability in that runtime compromises every workload.
